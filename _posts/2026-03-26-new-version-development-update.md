## 前言

最近把精力都扑在新版本的开发上，产品功能宣传暂时搁置了。一方面是时间确实不够用，另一方面是觉得与其宣传旧版本，不如让新用户直接上手新版本。

新版本的架构总算基本定型了——至少短期内不会再做大的调整，除非遇到不可抗力。这篇文章记录一下今天的几件流水账，顺便也算给自己留个存档。

---

## AI 工具选型：从闲鱼到 Codex

这几个月一直在探索更经济的 AI 编程方案。最早用的是某家 7 块 9 包月的服务，但用起来才发现，两天就能把一周的额度耗完。后来在闲鱼上发现了一个新选择——7.68 元可以买 3 天套餐，每天 200 美元的 Codex 额度，这几天基本够用了。

说实话，有点依赖症了。AI 辅助编程确实香，但成本控制也是个现实问题。闲鱼这个渠道不算正规，但算下来比官方订阅便宜不少，适合现阶段小团队精打细算。

---

## 编辑器抉择：Cursor 退租，回归 VS Code

Cursor 是个好工具，但有个问题让我实在受不了——打开状态时 CPU 占用经常飙到 90% 以上，Mac 明显卡顿。对于我这种同时开十几个窗口的人来说，这种卡顿很影响节奏。

忍了两周后，干脆退订了 20 美元的月费，编辑器切回 VS Code。VS Code 配合 GitHub Copilot，用起来也挺顺手，关键是资源占用可控。

这件事给我的教训是：**工具再好，如果吃系统资源影响到正常工作了，就得果断换**。工具是为人服务的，不能反过来让人迁就工具。

---

## 理想与现实：Web 到 App 的迁移困境

最初的设想是这样的：先完成 Web 与后端的对接，然后让 AI 直接参照 Web 的实现来改造 App。这样能复用已有的逻辑，开发效率更高。

实际干起来才发现，这个想法有点理想化了。主要的坑有两个：

**第一，App 和 Web 体验并不完全一致，功能也不完全兼容。**

有些在 Web 上很容易实现的功能，迁移到 iOS 上需要重新设计交互方式。AI 只能辅助，但不能替代产品思考。

**第二，AI 对于迁移改造，远没有新写功能来得顺畅。**

历史功能积累的坑太多——各种边界条件、隐藏逻辑、兼容代码——让 AI 去理解并适配这些内容，相较于从零开发一个新功能，反而更费时间。

这让我重新思考了一个问题：**什么时候该复用，什么时候该重写？** 并不是所有「重复工作」都该避免。当复用的成本超过重写时，重写反而是更理性的选择。

---

## 吹毛求疵的新动力

新版 UI 有了参照系，加上 AI 能力变强之后，做事的劲头反而更足了——也更「吹毛求疵」了。

以前觉得差不多能用的交互，现在会反复调整；突然想到一个小优化，就会立刻想动手去做。AI 让做事变快了，但对优先级的把握需要更精细。

这其实是把双刃剑：效率提升让我有能力追求更多细节，但同时也需要更强的自律来控制 scope creep（范围蔓延）。

---

## 璞奇启示

**第一，工具选择要匹配实际场景，而非追求「最好」。**

Cursor 确实是更好的工具，但在我这台 Mac 上跑不动，那就是不适合。璞奇的用户学习也是如此——最好的学习方式不是最先进的方式，而是最适合当前阶段和场景的方式。

**第二，复用与重写需要动态评估。**

初始方案往往基于理想假设，但实际执行中会遇到各种摩擦。有时候「重写」看起来是浪费，实际上是更低的长期成本。璞奇的练习设计也需要这种判断：什么时候该巩固已有知识，什么时候该引入新形式来重新激发学习者的兴趣。
